# Show Me — 个人作品与思考展示站

> Less but better. 让人安静地认识一个人。

## 零、风格参考

### 主参考：[paco.me](https://paco.me)

**喜欢的部分**：
- 极度克制的气质——纯文本驱动，像一封写得很好的信
- 单栏线性滚动，没有多余的视觉噪音
- 用斜体和文字节奏制造阅读韵律，不靠图片或动画
- Building / Projects / Writing / Now 的分区方式自然不造作
- 底部 manifesto 式的收尾（"Pray at the altar of hard work"）——有态度但不说教
- 暗色模式支持，黑白切换干净
- 整体像一个有品位的人安静地把东西摆在那里，不推销

**需要超越的部分**：
- 过分简单——作品只有标题 + 一行描述 + 链接，没有展开空间
- 每个项目值得被好好介绍，不能只是一个列表项
- 缺少视觉层次——全是文字，没有封面、截图、架构图的呼吸感
- 没有筛选/分类机制，作品多了之后会变成一长串
- 缺少叙事深度——看完知道他做了什么，但不知道为什么做、怎么想的

**设计方向**：
> paco.me 的克制与文字质感作为基底，但在作品展示上给每个项目一个值得停留的空间。
> 文字为主，视觉为辅——但「辅」不是「无」，而是在对的地方出现对的画面。

## 一、这个网站是什么

不是传统简历，不是博客，不是作品集模板。

这是一个**活的空间**——展示 Anner 在做什么、在想什么、在构建什么。访客进来后，能感受到一个人的质地：技术深度、审美取向、思考方式、和对世界的态度。

核心隐喻：**一间私人工作室的开放日**。你走进来，看到墙上挂着的作品，桌上摊开的笔记，书架上的痕迹。没有导游喊话，但一切都在说话。

---

## 二、设计原则

### 2.1 内容会生长

项目会不断增加，作品会不断产出。网站架构必须是**可扩展的内容容器**，而非固定页数的静态站。新增一个作品，应该像往书架上放一本书一样自然——不需要重新装修房间。

### 2.2 每个作品值得被好好介绍

不是缩略图网格 + 一行标题。每个作品都有自己的故事：为什么做、怎么做、学到了什么。作品详情页是一等公民，不是附属页面。

### 2.3 交互有巧思但不花哨

- 动画服务于叙事，不服务于炫技
- 微交互让人会心一笑，不让人等待
- 滚动是主要的叙事手段——像翻书，不像坐过山车
- 转场平滑自然，像呼吸，不像闪光灯

### 2.4 温暖的未来感

遵循 CLAUDE.md 中定义的设计语言：大地色系、柔和光感、有肌理的留白。拒绝冷金属、赛博朋克、霓虹闪烁。

## 三、信息架构

```
Landing (首屏)
│
├── About (关于我)
│   └── 简短的自我叙述，不是条目式简历
│
├── Works (作品)
│   ├── 作品卡片流（可筛选、可排序）
│   └── 作品详情页（每个作品一个独立叙事空间）
│
├── Thinking (思考)
│   └── 文章/笔记/随想（按主题聚类，不按时间线）
│
├── Journey (旅程)
│   └── 时间线视角：职业经历、关键节点、成长轨迹
│
└── Connect (联系)
    └── 简洁的联系方式
```

### 为什么是这五个区域

| 区域 | 回答的问题 | 不是什么 |
|------|-----------|---------|
| Landing | 这个人给我什么感觉？ | 不是信息轰炸，是氛围建立 |
| About | 他是谁？在做什么？ | 不是流水账履历，是一段真诚的自述 |
| Works | 他造了什么？ | 不是截图画廊，是每个作品的故事 |
| Thinking | 他怎么想问题？ | 不是博客归档，是思维的切面 |
| Journey | 他怎么走到今天的？ | 不是年份列表，是有节奏的叙事 |

## 四、页面细节

### 4.1 Landing — 第一印象

**目标**：3 秒内建立氛围，不说教，不自夸。

- 一句话定位（不是 slogan，是一种自我描述的温度）
- 候选方向：
  - `构建工具，记录思考。`
  - `在企业平台与独立创造之间。`
  - 安静的打字机效果，或者墨水晕染般渐显
- 背景：微妙的粒子/纹理动画，像纸张的纤维在呼吸
- 向下滚动有明确的引导，但不是箭头——可以是文字渐显，可以是内容的边缘露出来

**交互**：
- 页面加载时，元素依次安静地出现（opacity + 轻微位移），像清晨房间里的光线慢慢亮起来
- 鼠标移动时，背景纹理有极微弱的视差响应（1-2px 级别），营造空间感

### 4.2 About — 关于我

**目标**：让人在 30 秒内理解你是一个什么样的人。

- 不用条目式罗列技能栈
- 用 2-3 段自然语言讲清楚：做什么工作、业余在造什么、关心什么问题
- 技能/工具以标签云或散落的方式呈现，不是进度条
- 可以有一张生活化的照片（不是证件照），或者干脆用一个抽象的视觉符号代替

**关键内容素材**：
- 帆软平台工程师：权限、调度、集群、缓存——企业级系统的地基
- 独立创造者：Eden、Northstar、NotionHub、OpenOctopus、MCP 生态贡献
- AI 原生开发者：RAG、Agent 设计模式、Vibe Coding 实践者
- 思考者：第一性原理、工程文化、人生长远规划

### 4.3 Works — 作品

**这是网站的核心，也是会持续生长的部分。**

#### 列表页

- 卡片式布局，每张卡片是一个作品的入口
- 卡片内容：项目名 + 一句话描述 + 技术标签 + 封面视觉
- 支持按类别筛选：`工具` `平台` `AI` `开源贡献` `实验`
- 卡片 hover 时有微妙的提升感（shadow + scale），像从桌面上拿起一张卡片
- **新作品出现时不需要改布局**——卡片流自动容纳

#### 详情页（重点）

每个作品详情页是一个**独立的叙事空间**，结构如下：

```
作品名
一句话：这是什么

---

背景与动机
为什么要做这个？解决什么问题？

---

方案与实现
怎么做的？关键技术决策是什么？
（配图：架构图 / 截图 / 代码片段）

---

收获与反思
学到了什么？如果重来会怎么做？

---

技术栈标签
链接（源码 / 在线体验 / 相关文章）
```

**交互**：
- 详情页采用长滚动叙事，每个章节之间有安静的分隔——不是分割线，是留白和节奏
- 图片/截图在滚动到时缓缓显现（fade-in + 轻微上浮）
- 代码块有语法高亮，但配色融入页面整体色调（暖色系代码主题）
- 页面右侧可以有一个极简的章节导航（小圆点 / 短横线），hover 时显示章节名

### 4.4 Thinking — 思考

**目标**：展示思维方式，不是堆文章数量。

- 按主题聚类，不按发布时间排列
- 主题方向：
  - `工程哲学` — 第一性原理、拒绝过度设计、工程文化
  - `AI 实践` — Vibe Coding、RAG、Agent 设计
  - `职业成长` — 从技术专家到系统思考者的转型
  - `生活观察` — 人生长远思考、帆软商业观察
- 每篇文章入口：标题 + 一句摘要 + 主题标签
- 文章阅读页面：舒适的阅读排版，行宽适中（60-75 字符），行高宽松

**交互**：
- 主题切换时，卡片有轻柔的重排动画（layout animation），像书架上的书在重新分类
- 文章页面有阅读进度指示（顶部细线），安静但有用

### 4.5 Journey — 旅程

**目标**：用叙事而非表格讲述职业路径。

- 不是传统的纵向时间线
- 而是**横向滚动的旅程**（或纵向滚动但有明确的章节感）
- 每个阶段是一张"卡片"或一个"房间"：
  - 时间段
  - 角色 / 做什么
  - 关键事件 / 里程碑
  - 那个阶段的一句感悟
- 节点之间有连线或过渡效果，暗示因果关系而非仅仅时间顺序

**交互**：
- 滚动驱动的叙事（scroll-driven narrative）
- 每个阶段进入视口时，内容依次显现
- 时间线上的节点可以点击展开更多细节

### 4.6 Connect — 联系

- 极简，不需要表单
- GitHub / Email / 其他你愿意公开的联系方式
- 可以有一句话：`如果你觉得我们可能聊得来`
- 链接 hover 时有温暖的微交互（下划线展开 / 颜色渐变）

## 五、视觉系统

### 5.1 色彩

```
背景色：   #FAF8F5（暖白，像未漂白的纸）
主文字色：  #2C2C2C（深炭灰，不是纯黑）
次文字色：  #7A7268（暖灰）
强调色：   #C4956A（柔和琥珀）
辅助色：   #8B9E82（鼠尾草绿）
代码背景：  #F0EDE8（暖米色）
卡片背景：  #FFFFFF（纯白，用阴影区分层次）
```

### 5.2 字体

```
标题：    思源黑体 / Noto Sans SC（中文）
         Space Grotesk / Inter（英文）
正文：    思源宋体 / Noto Serif SC（中文长文阅读）
         Inter / Source Serif 4（英文）
代码：    JetBrains Mono / Fira Code
```

### 5.3 间距与节奏

- 大量留白，内容区域最大宽度 720px（阅读页）/ 1200px（卡片页）
- 章节之间用 120-160px 的留白分隔，不用分割线
- 元素之间的间距遵循 8px 网格

### 5.4 动画原则

```
时长：    200-400ms（微交互）/ 600-800ms（页面转场）
缓动：    ease-out 为主（自然减速感）
触发：    滚动进入视口 / hover / 页面切换
禁止：    弹跳、闪烁、抖动、持续循环动画
心法：    如果用户注意到了动画本身，那就做过头了
```

## 六、技术选型

| 层面 | 选择 | 理由 |
|------|------|------|
| 框架 | Astro + React Islands | 前端 SSG + 需要交互的地方用 React，后台管理和 API 共存 |
| 样式 | Tailwind CSS | 与现有项目栈一致 |
| 数据库 | SQLite + Drizzle ORM | 零运维，单文件，个人站量级足够 |
| MCP | @modelcontextprotocol/sdk (TypeScript) | 官方维护最活跃，与 Node 生态一致 |
| 认证 | Session Cookie (Admin) + Bearer Token (MCP) | 两种场景两种方案 |
| 动画 | Framer Motion + CSS 原生动画 | 声明式动画，不重 |
| 部署 | Vercel / Cloudflare Pages | 零运维，全球 CDN |
| 图片 | 本地 + Astro Image 优化 | 自动 WebP、响应式 |

### 架构总览

```
┌──────────────────────────────────────────────┐
│                  Astro App                    │
│                                               │
│  /             → 前端展示页面（公开，SSG）      │
│  /admin        → 管理看板（Session Cookie）    │
│  /api/auth     → 登录/登出接口                 │
│  /api/mcp      → MCP Streamable HTTP 端点     │
│                                               │
│  ┌───────────────────────────────────┐        │
│  │  Service Layer（共享业务逻辑）      │        │
│  │  works / articles / journey /     │        │
│  │  profile / contacts / settings    │        │
│  └──────────────┬────────────────────┘        │
│                 │                              │
│  ┌──────────────▼────────────────────┐        │
│  │  SQLite + Drizzle ORM             │        │
│  └───────────────────────────────────┘        │
└──────────────────────────────────────────────┘
```

- **一个项目**：前端展示、管理看板、MCP 端点全在同一个 Astro 项目中
- **Service Layer** 是核心：MCP tools 和 Admin 页面都调用同一层逻辑
- 前端页面构建时从 SQLite 读数据，SSG 输出静态页面

## 八、MCP Server

### 8.1 传输与安全

- **传输方式**：Streamable HTTP，端点 `/api/mcp`
- **认证**：Bearer Token，通过环境变量 `MCP_AUTH_TOKEN` 配置
- **权限**：所有工具都需要有效 Token，无效/缺失返回 401
- **限流**：Rate limiting 防滥用

### 8.2 工具清单（30 个）

```
── Works (作品) ──────────────────────
list_works          筛选/排序作品列表
get_work            获取单个作品详情
create_work         创建作品
update_work         更新作品字段
delete_work         删除作品
reorder_works       调整作品排序
set_work_featured   设置/取消首页推荐

── Articles (文章) ──────────────────
list_articles       筛选/排序文章列表
get_article         获取单篇文章（含正文）
create_article      创建文章
update_article      更新文章
delete_article      删除文章
publish_article     发布/取消发布

── Journey (旅程) ──────────────────
list_stages         获取所有阶段
get_stage           获取阶段详情（含事件）
create_stage        创建阶段
update_stage        更新阶段
delete_stage        删除阶段
create_event        阶段内添加事件
update_event        更新事件
delete_event        删除事件
reorder_stages      调整阶段排序

── Profile (个人信息) ──────────────
get_profile         获取个人信息
update_profile      更新个人信息

── Contacts (联系方式) ─────────────
list_contacts       获取联系方式列表
create_contact      添加联系方式

delete_contact      删除联系方式

── Site (站点设置) ─────────────────
get_settings        获取所有设置
update_setting      更新单个设置项

── Resume (简历导出) ──────────────
list_templates      获取可用简历模板
preview_resume      预览简历（返回 HTML）
export_resume_pdf   导出简历为 PDF
```

## 九、管理看板

### 9.1 定位

只读看板，不做编辑。所有内容修改通过 MCP 接口完成。
后台的价值是「一眼看到所有内容的状态」。

### 9.2 路由

```
/admin/login   → 登录页（用户名 + 密码）
/admin         → 看板主页（登录保护）
```

### 9.3 认证

- 用户名密码通过环境变量配置（`ADMIN_USERNAME` / `ADMIN_PASSWORD`）
- 登录后生成 Session Cookie，httpOnly + secure
- 未登录访问 `/admin` 重定向到 `/admin/login`

### 9.4 整体布局

左侧导航栏（200px） + 右侧内容区：

**导航栏分组**：
- 导航：看板总览、作品、文章、旅程、个人信息、联系方式、站点设置
- 工具：简历导出

**看板总览页**：
- 概览行：作品数、文章数（已发布/草稿）、旅程阶段数、联系方式数
- 作品列表：标题 + 状态标签 + 标签 + 是否推荐
- 文章列表：标题 + 主题 + 发布状态 + 更新时间
- 旅程时间线：年份 → 阶段标题，横向展示
- 个人信息 + 联系方式

### 9.5 简历导出页

基于站点数据自动生成简历，支持预览和 PDF 导出：

- **模板选择器**：缩略图卡片，支持切换不同简历模板
  - 经典（单栏，传统布局）
  - 双栏（左侧深色侧栏 + 右侧内容）
  - 极简（居中对齐，大留白）
- **A4 预览区**：居中展示模拟纸张（白底 + 投影），实时预览选中模板效果
- **操作按钮**：刷新预览 + 导出 PDF
- 所有内容从数据库自动填充（个人信息、经历、作品、技能）

配色沿用前端暖色体系，略微中性化（更「工具感」）。

## 十一、成功标准

这个网站做对了，如果：

1. **一个陌生人花 3 分钟浏览后，能说出你是做什么的、擅长什么、关心什么**
2. **通过 MCP 对 Claude 说「帮我添加一个新作品」，内容就出现在网站上**
3. **每个作品详情页让人想读完，而不是扫一眼就走**
4. **管理看板打开就能看到所有内容的状态，不需要翻页**
5. **整体感受是安静、有质感、值得停留的**——不是又一个千篇一律的开发者主页
6. **一年后回来看，依然觉得好看**
